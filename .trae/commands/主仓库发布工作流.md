## 🎯 任务：主仓库发布工作流

> ⚠️ **适用场景**：此工作流仅适用于主仓库（origin）。衍生仓库请使用 `衍生版发布工作流`。
>
> 📌 **核心设计**：
> - 主仓库是底座更新的唯一来源
> - 主仓库内容优先级最高，强制覆盖衍生仓库
> - `配置/站点配置` 目录是衍生仓库的独有内容，不被覆盖

### 🔄 步骤0：拉取所有远程仓库最新状态 📥

> ⚠️ **重要**：这是新增的关键步骤，确保在发布前获取所有远程仓库的最新状态

- ☁️ 获取所有远程仓库列表：
  ```
  sudo -u www git remote
  ```
- 📥 **从所有远程仓库拉取最新状态**：
  ```
  sudo -u www git fetch --all
  ```
- 🔍 检查是否有远程仓库比本地更新：
  ```
  sudo -u www git status
  ```
- 📋 如果有远程仓库更新，需要先决策：
  - **如果远程有新提交**：需要评估是否需要合并或强制使用本地版本
  - **如果远程落后**：继续后续步骤，强制推送覆盖

### 📋 步骤1：修改更新日志 ✏️
- 📂 打开 `CHANGELOG.md` 或 `HISTORY.md` 文件
- 📅 使用 `TZ='Asia/Shanghai' date '+%Y-%m-%d %H:%M'` 获取中国时区当前日期时间
- ➕ 在文件顶部添加新的版本记录，包含：
  - 🔖 版本号（如 v1.2.3）
  - 📅 发布日期（YYYY-MM-DD HH:MM 格式，中国时区 Asia/Shanghai）
  - 🏷️ 本次更新的内容分类（新增/变更/修复/移除/安全）
  - 📝 具体的变更描述
- 🎨 **颜表情装饰**：为更新内容添加合适的颜表情，提升可读性

### 📄 步骤2：同步更新 README.md 版本信息 🔄
- 📂 打开 `README.md` 文件
- 🔍 找到版本徽标（格式如 `![版本](https://img.shields.io/badge/版本-vX.X.X-blue?style=flat-square)`）和最后更新时间徽标（格式如 `![最后更新](https://img.shields.io/badge/最后更新-YYYY.MM.DD%20HH:MM-orange?style=flat-square)`）
- ✏️ 只修改这两个徽标，更新其中的版本号和日期时间，使其与 CHANGELOG.md 中的最新版本保持一致
- 🎯 确保日期时间格式为 `YYYY.MM.DD%20HH:MM`（空格用 %20 代替，中国时区 Asia/Shanghai），版本号格式为 `vX.X.X`
- 💡 如果没有具体时间，时间部分可以为空，格式为 `YYYY.MM.DD`
- 💡 其他徽标（访客数量、GitHub Stars、Forks、Contributors、Issues、Last Commit、Code Size、开发语言、平台、脚本环境 等）不需要修改

### 📋 步骤3：分析代码变更 🔍
- 📂 使用 `git status` 查看当前修改的文件
- 📝 使用 `git diff` 查看具体的代码变更内容
- 🧠 基于变更内容，AI自动分析并总结修改了什么

### 📋 步骤4：生成提交信息 ✍️
- 📅 使用 `TZ='Asia/Shanghai' date '+%Y-%m-%d %H:%M:%S'` 获取中国时区当前日期时间
- 🤖 AI根据代码变更自动生成提交标题，要求：
  - 🔖 包含变更类型（新增/修改/修复/重构/文档/样式 等）
  - 📝 简明扼要概括核心变更
  - 📅 在标题末尾附加日期时间，格式：`[YYYY-MM-DD HH:MM:SS]`（中国时区）
- 📄 AI根据代码变更自动生成提交正文，要求：
  - 📋 列出具体修改了哪些文件
  - 🔍 描述每个文件修改了什么内容
  - 🏷️ 按变更类型分类说明
- 💡 提交信息格式示例：
  ```
  🔧 修复配置文件解析错误 [2025-05-28 15:30:00]

  - 修改了 nginx.conf 中 server 块的解析逻辑
  - 更新了日志轮转配置参数
  - 修复了端口冲突检测问题
  ```

### 🚀 步骤5：使用www用户提交代码 💾
- ➕ 使用 `git add .` 或 `git add -A` 添加所有修改的文件
- 👤 使用以下命令以www用户身份提交：
  ```
  sudo -u www git commit --author='www <www@localhost>' -m 'AI生成的提交标题' -m 'AI生成的提交正文'
  ```

### 🔄 步骤6：强制推送到所有远程仓库（覆盖衍生仓库） 📤

> ⚠️ **核心逻辑**：主仓库内容强制覆盖所有衍生仓库，确保底座完全同步

- ☁️ 首先获取当前分支名：
  ```
  sudo -u www git branch --show-current
  ```
- ☁️ 获取所有远程仓库列表：
  ```
  sudo -u www git remote
  ```
- ☁️ **推送到所有远程仓库**（强制覆盖，排除站点配置目录）：
  ```
  # 示例：假设 git remote 输出为：衍生仓库A, 衍生仓库B, origin
  # 则需要执行以下推送命令（以www用户身份）：
  sudo -u www git push origin <分支名>
  sudo -u www git push 衍生仓库A <分支名> --force
  sudo -u www git push 衍生仓库B <分支名> --force
  ```
- ⚠️ **重要**：必须遍历 `git remote` 输出的每个远程仓库名称，逐个执行推送命令
- 💡 **推送逻辑**：
  1. 先推送到 origin（主仓库）
  2. 再推送到所有衍生仓库，使用 `--force` 强制推送
  3. 主仓库内容优先级最高，强制覆盖衍生仓库的除站点配置外的所有内容
- ⚠️ 注意：使用 sudo -u www 执行推送命令

### 📌 关键说明

#### 🎯 主仓库优先级原则
- 主仓库是底座更新的唯一权威来源
- 主仓库的任何更新都会强制同步到所有衍生仓库
- 衍生仓库的本地修改（除站点配置外）会被主仓库覆盖

#### 🔒 独有文件保护机制
- `配置/站点配置/本地站点.conf` - 被 `.gitignore` 忽略，不会被推送
- `配置/站点配置/*.conf` - 每个衍生仓库独有的站点配置，不会被主仓库覆盖
- `日志/*.log` - 日志文件，不会被推送
- 这些独有文件只存在于本地，不会影响仓库间的同步

#### 📊 仓库角色说明
| 仓库类型 | 仓库名 | 操作权限 | 同步策略 |
|---------|--------|---------|---------|
| 主仓库 | origin | 拉取 + 推送 | 正常推送（非强制） |
| 衍生仓库 | 其他远程仓库 | 仅推送 | 强制推送覆盖（排除站点配置） |

#### 🔄 工作流程图
```
主仓库更新
    ↓
修改版本文件（CHANGELOG.md、README.md）
    ↓
提交变更
    ↓
推送到 origin（主仓库）
    ↓
强制推送到所有衍生仓库（覆盖除站点配置外的所有内容）
    ↓
完成同步 ✅
```
