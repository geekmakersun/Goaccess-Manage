## 🎯 任务：衍生版发布工作流

> ⚠️ **适用场景**：此工作流仅适用于衍生仓库。主仓库请使用 `主仓库发布工作流`。
>
> 📌 **核心设计**：
> - 衍生版首先拉取主仓库的最新更新（远程优先）
> - 只推送衍生仓库独有的变更（站点配置等）
> - 不修改版本文件（README.md、CHANGELOG.md）
> - 主仓库内容优先级最高，覆盖衍生仓库的底座文件

### 🔄 步骤0：从主仓库拉取最新更新 📥

> ⚠️ **重要**：这是关键步骤，确保衍生仓库同步主仓库的最新底座

- ☁️ 获取当前分支名：
  ```
  sudo -u www git branch --show-current
  ```
- 📥 **从主仓库（origin）拉取最新更新**：
  ```
  sudo -u www git pull origin <当前分支名>
  ```
- 🔍 检查是否有冲突：
  ```
  sudo -u www git status
  ```
- 📋 如果有冲突（通常只会在站点配置目录外）：
  - **优先使用远程版本**：主仓库的内容优先级最高
  - **解决冲突**：使用 `sudo -u www git checkout --theirs <文件>` 采用远程版本
  - **保留独有文件**：`配置/站点配置` 目录下的文件不会被覆盖
- ✅ 确认同步完成后再继续后续步骤

### 📋 步骤1：分析本地独有变更 🔍

> ⚠️ **重要**：只分析衍生仓库独有的变更，不包括从主仓库同步的内容

- 📂 使用 `git status` 查看当前修改的文件
- 📝 检查修改的文件是否在以下目录：
  - ✅ **允许修改**：`配置/站点配置/*.conf`（站点独有配置）
  - ✅ **允许修改**：其他衍生仓库独有的定制文件
  - ❌ **不应修改**：`README.md`、`CHANGELOG.md`（由主仓库管理）
  - ❌ **不应修改**：`脚本/*.sh`、`文档/*.md`（底座文件，由主仓库管理）
- 🧠 基于变更内容，AI自动分析并总结修改了什么

### 📋 步骤2：生成提交信息 ✍️
- 📅 使用 `TZ='Asia/Shanghai' date '+%Y-%m-%d %H:%M:%S'` 获取中国时区当前日期时间
- 🤖 AI根据代码变更自动生成提交标题，要求：
  - 🔖 包含变更类型（新增/修改/修复/重构/配置 等）
  - 📝 简明扼要概括核心变更（强调这是衍生仓库的独有变更）
  - 📅 在标题末尾附加日期时间，格式：`[YYYY-MM-DD HH:MM:SS]`（中国时区）
- 📄 AI根据代码变更自动生成提交正文，要求：
  - 📋 列出具体修改了哪些文件
  - 🔍 描述每个文件修改了什么内容
  - 🏷️ 明确标注这些是衍生仓库的独有变更
- 💡 提交信息格式示例：
  ```
  🔧 更新站点配置（衍生仓库独有） [2025-05-28 15:30:00]

  - 配置了本地站点的日志分析参数
  - 调整了报告输出路径
  - 添加了站点特有的自定义配置
  ```

### 🚀 步骤3：使用www用户提交代码 💾
- ➕ 使用 `git add .` 或 `git add -A` 添加所有修改的文件
- 👤 使用以下命令以www用户身份提交：
  ```
  sudo -u www git commit --author='www <www@localhost>' -m 'AI生成的提交标题' -m 'AI生成的提交正文'
  ```

### 📤 步骤4：推送到衍生仓库的远程仓库 📤

> ⚠️ **重要**：衍生版只推送到自己的远程仓库，不推送到其他仓库

- ☁️ 获取当前分支名：
  ```
  sudo -u www git branch --show-current
  ```
- ☁️ 获取当前仓库名称（用于识别这是哪个衍生版）：
  ```
  sudo -u www basename $(git remote get-url origin 2>/dev/null || echo 'unknown') .git
  ```
- ☁️ **只推送到当前衍生版的远程仓库**，以www用户身份执行：
  ```
  sudo -u www git push origin <当前分支名>
  ```
- ⚠️ **重要**：衍生版只推送到自己的 origin 远程仓库，**不推送到其他远程仓库**
- 🎯 **设计理念**：主仓库负责更新底座并推送到所有衍生版，衍生版只负责推送自己的定制内容

### 📌 关键说明

#### 🎯 衍生仓库优先级原则
- **主仓库内容优先级最高**：底座文件（脚本、文档、配置模板等）以主仓库为准
- **远程优先原则**：从主仓库拉取更新时，主仓库内容覆盖本地
- **独有文件保护**：`配置/站点配置` 目录下的文件不会被主仓库覆盖

#### 🔒 独有文件保护机制
- `配置/站点配置/本地站点.conf` - 被 `.gitignore` 忽略，不会被推送
- `配置/站点配置/*.conf` - 每个衍生仓库独有的站点配置，不会被主仓库覆盖
- `日志/*.log` - 日志文件，不会被推送
- 这些独有文件只存在于本地，是衍生仓库的个性化配置

#### 📊 仓库角色说明
| 仓库类型 | 仓库名 | 操作权限 | 同步策略 |
|---------|--------|---------|---------|
| 主仓库 | origin | 拉取 + 推送 | 拉取主仓库更新（优先级最高），推送独有变更 |
| 其他远程仓库 | - | 不涉及 | 衍生版不推送到底座仓库 |

#### 🔄 工作流程图
```
衍生仓库工作流
    ↓
从主仓库拉取最新更新（远程优先）
    ↓
检查并解决冲突（优先使用远程版本）
    ↓
修改站点配置等独有文件
    ↓
提交独有变更
    ↓
推送到衍生仓库的远程仓库
    ↓
完成同步 ✅
```

#### ⚠️ 注意事项

1. **不要修改版本文件**：
   - ❌ 不要修改 `README.md` 中的版本号
   - ❌ 不要修改 `CHANGELOG.md`
   - ✅ 这些文件由主仓库统一管理

2. **不要修改底座文件**：
   - ❌ 不要修改 `脚本/*.sh`（如需修改，应在主仓库进行）
   - ❌ 不要修改 `文档/*.md`（如需修改，应在主仓库进行）
   - ❌ 不要修改 `配置/配置模板.conf`（如需修改，应在主仓库进行）
   - ✅ 只修改 `配置/站点配置/*.conf` 等独有配置文件

3. **同步时机**：
   - ✅ 建议在每次修改前先从主仓库拉取最新更新
   - ✅ 定期同步主仓库的更新，保持底座一致性

#### 💡 衍生仓库使用场景

- **场景1：配置站点独有参数**
  - 修改 `配置/站点配置/本地站点.conf`
  - 提交并推送到衍生仓库的远程仓库

- **场景2：主仓库发布了新版本**
  - 执行步骤0：从主仓库拉取最新更新
  - 底座文件自动更新，站点配置保持不变

- **场景3：需要修改底座功能**
  - 不应在衍生仓库直接修改
  - 应在主仓库修改后发布，衍生仓库拉取更新
