## 🎯 任务：完整发布工作流

### 📋 步骤1：修改更新日志 ✏️
- 📂 打开 `CHANGELOG.md` 或 `HISTORY.md` 文件
- 📅 使用 `git log -1 --format="%Y-%m-%d %H:%M"` 获取发布日期（与提交时间一致）
- ➕ 在文件顶部添加新的版本记录，包含：
  - 🔖 版本号（如 v1.2.3）
  - 📅 发布日期（YYYY-MM-DD HH:MM 格式，使用上一步获取的时间）
  - 🏷️ 本次更新的内容分类（新增/变更/修复/移除/安全）
  - 📝 具体的变更描述
- 🎨 **颜表情装饰**：为更新内容添加合适的颜表情，提升可读性

### 📄 步骤2：同步更新 README.md 版本信息 🔄
- 📂 打开 `README.md` 文件
- 🔍 找到版本徽标（格式如 `![Version](https://img.shields.io/badge/version-vX.X.X-blue?style=flat-square)`）和最后更新时间徽标（格式如 `![Last Update](https://img.shields.io/badge/last%20update-YYYY.MM.DD%20HH:MM-orange?style=flat-square)`）
- ✏️ 只修改这两个徽标，更新其中的版本号和日期时间，使其与 CHANGELOG.md 中的最新版本保持一致
- 🎯 确保日期时间格式为 `YYYY.MM.DD%20HH:MM`（空格用 %20 代替），版本号格式为 `vX.X.X`
- 💡 其他徽标（访客数量、GitHub Stars、Forks、Contributors、Issues、Last Commit、Code Size、Language、Platform、Shell 等）不需要修改

### 📋 步骤3：分析代码变更 🔍
- 📂 使用 `git status` 查看当前修改的文件
- 📝 使用 `git diff` 查看具体的代码变更内容
- 🧠 基于变更内容，AI自动分析并总结修改了什么

### 📋 步骤4：生成提交信息 ✍️
- 📅 使用 `date '+%Y-%m-%d %H:%M:%S'` 获取当前日期时间
- 🤖 AI根据代码变更自动生成提交标题，要求：
  - 🔖 包含变更类型（新增/修改/修复/重构/文档/样式 等）
  - 📝 简明扼要概括核心变更
  - 📅 在标题末尾附加日期时间，格式：`[YYYY-MM-DD HH:MM:SS]`
- 📄 AI根据代码变更自动生成提交正文，要求：
  - 📋 列出具体修改了哪些文件
  - 🔍 描述每个文件修改了什么内容
  - 🏷️ 按变更类型分类说明
- 💡 提交信息格式示例：
  ```
  🔧 修复配置文件解析错误 [2025-05-28 15:30:00]

  - 修改了 nginx.conf 中 server 块的解析逻辑
  - 更新了日志轮转配置参数
  - 修复了端口冲突检测问题
  ```

### 🚀 步骤5：使用www用户提交代码 💾
- ➕ 使用 `git add .` 或 `git add -A` 添加所有修改的文件
- 👤 使用以下命令以www用户身份提交：
  ```
  git commit --author="www <www@localhost>" -m "AI生成的提交标题" -m "AI生成的提交正文"
  ```
- ☁️ 首先获取当前分支名：
  ```
  git branch --show-current
  ```
- ☁️ 获取所有远程仓库列表：
  ```
  git remote
  ```
- ☁️ 挨个推送到所有远程仓库，以www用户身份执行：
  ```
  su - www -c "git push origin <当前分支名>"
  su - www -c "git push <远程仓库名1> <当前分支名>"
  su - www -c "git push <远程仓库名2> <当前分支名>"
  ```
- ⚠️ 注意：不要使用sudo，应该使用su切换到www用户执行推送命令
- 📌 **仓库角色说明**：
  - `origin` - 主仓库，支持拉取(fetch)、合并(merge)和推送(push)
  - `<远程仓库名1>` 和 `<远程仓库名2>` - 只推送仓库，仅支持推送(push)，无法拉取
